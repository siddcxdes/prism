# Prism Architecture & Internals

This document explains how Prism is built, how its parts work together, and what happens inside the system when a user asks something.

---

# 1. High-Level Overview

Prism is divided into two main parts:

## Frontend (React + Vite)

This is the part that the user sees and interacts with.

It is responsible for:
- Showing dashboards
- Handling login and signup
- Displaying charts and research reports
- Giving users a clean and simple interface

## Backend (FastAPI)

This is the main logic layer of the application.

It is responsible for:
- Receiving requests from the frontend
- Saving and fetching data from the database
- Running AI-based research pipelines
- Handling authentication and security
- Returning processed results back to the frontend

## Main Purpose of Prism

The most important feature of Prism is its Agentic Research Pipeline.

When a user asks a question such as:

"How is Tesla doing compared to Rivian?"

Prism follows a complete process:
1. It understands the user query
2. It searches for relevant information
3. It fetches stock and company data
4. It combines everything together
5. It uses AI to generate a final research report

# 2. Infrastructure and Deployment

Prism uses different services for hosting and deployment.

## Frontend Hosting

The frontend is hosted on Vercel.

Vercel:
- Automatically builds the React application
- Deploys the latest changes
- Handles frontend routing using `react-router`
- Uses `vercel.json` for route rewrites

## Backend Hosting

The backend is hosted on Render as a Python web service.

The backend:
- Runs using `uvicorn`
- Handles API requests
- Connects with the database
- Runs the AI research pipeline

## Database

Prism uses PostgreSQL through Neon.

Neon is a serverless PostgreSQL service.

The database is connected through SQLAlchemy ORM, which allows Python classes to interact with database tables.

## LLM Provider

Prism uses Groq Cloud API for running Large Language Models.

The model being used is Meta LLaMA 3.3 70B.

This model is used because:
- It is very fast
- It gives good quality responses
- It is useful for creating research summaries and insights

# 3. Database Schema

Prism uses a relational database design.

There are four main tables in the system:

## Organisations Table

This table stores organisation details.

An organisation is basically a group of users working together.

Each organisation has:
- A unique ID
- An invite code
- Its own isolated data

This is useful because different companies or teams should not see each other's data.

## Users Table

This table stores user information.

Each user belongs to one organisation.

It stores:
- User name
- Email
- Password hash
- User role such as Admin or Analyst
- Organisation ID

Passwords are never stored directly. They are hashed using bcrypt.

## Research Table

This is one of the most important tables in Prism.

It stores the research reports generated by the AI.

Each research record can contain:
- User query
- Summary
- Sentiment
- Trends
- Insights
- Stock data
- Confidence score
- Raw AI response

## Watchlist Table

This table is used to save companies or stocks that users want to track.

For example, a user may want to keep watching:
- Tesla
- Rivian
- Apple
- Nvidia

The watchlist helps users quickly return to important companies without searching again.

# 4. The Agentic AI Pipeline

The AI pipeline is the main engine of Prism.

It is located inside:

`backend/app/ai/groq_client.py`

The pipeline works in three phases.

## Phase 1: Planning

When the user enters a question, Prism first sends that question to the LLM with a planning prompt.

The job of the LLM here is not to answer the question directly.

Instead, it decides:
- Which tools are needed
- Which companies or stock tickers are involved
- What data should be collected

For example, if the user asks:

"How is Tesla doing compared to Rivian?"

The LLM may return a JSON response like this:

```json
{
  "tools": ["search_news", "fetch_stock_data"],
  "stock_tickers": ["TSLA", "RIVN"]
}
